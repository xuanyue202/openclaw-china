# 微信客服接入 OpenClaw 的开发背景

## 1. 目标

本项目希望将企业微信「微信客服」能力接入 OpenClaw，使外部微信用户可以通过微信客服入口与 OpenClaw 绑定的 Agent 进行对话。

接入完成后，预期支持以下基本场景：

- 微信用户进入客服会话后，消息能够进入 OpenClaw 的统一路由与回复体系
- OpenClaw 生成的回复能够通过微信客服接口回发给用户
- 在满足平台条件时，支持发送欢迎语或事件响应消息
- 后续可以逐步扩展客户信息补充、客服入口生成、多账号、媒体消息等能力

## 2. 为什么要做独立的 `wecom-kf` 渠道

微信客服与现有的企业微信智能机器人、自建应用不是同一套对象模型和接口体系。

它们的核心差异如下：

- 面向对象不同：微信客服面向外部微信用户，现有 `wecom` / `wecom-app` 更偏企业内部成员、群会话或企业微信侧会话
- 身份模型不同：微信客服核心标识是 `open_kfid`、`external_userid`、客服会话；不是 `agentid`、`userid`、`botId`
- 接入模型不同：微信客服需要同时处理回调验签、解密、事件通知、`sync_msg` 拉消息、`send_msg` 回消息
- 平台限制不同：微信客服发送受 48 小时窗口和消息条数限制约束

因此，这个能力不适合直接复用当前的 `wecom` 或 `wecom-app` 渠道，而应作为一个新的 OpenClaw 渠道实现，推荐渠道 ID 为 `wecom-kf`。

## 3. 官方文档给出的能力边界

从当前已导入的官方文档可以确认，微信客服已经提供实现 OpenClaw 接入所需的完整最小闭环：

1. 通过 `corpId + corpSecret` 获取 `access_token`
2. 配置回调地址、`Token`、`EncodingAESKey`
3. 接收回调事件
4. 通过 `sync_msg` 拉取实际消息和事件
5. 通过 `send_msg` 回发客服消息
6. 通过 `send_msg_on_event` 发送欢迎语等事件响应消息

这说明从 API 能力角度，接入 OpenClaw 没有明显缺口。

## 4. 需要特别理解的官方约束

### 4.1 回调不是完整消息体

微信客服的回调更像“有新消息/事件”的通知，而不是直接把完整消息内容推给开发者。

实际流程是：

1. 微信客服回调企业自建的 URL
2. 企业完成验签和解密
3. 从回调中拿到 `token` 和可选的 `open_kfid`
4. 再调用 `sync_msg` 拉取真实消息列表
5. 用 `cursor` 持续增量消费后续消息

这意味着插件必须自己实现：

- 回调 GET/POST 处理
- `sync_msg` 拉取
- `cursor` 持久化
- 消息去重

### 4.2 发送存在平台级限制

微信客服的主动回发能力并非无限制：

- 用户主动发消息后，企业可在 48 小时内调用发送接口
- 当前平台限制为最多可下发 5 条消息
- 欢迎语不是普通消息发送，需要通过事件返回的 `welcome_code` 在限定时间内调用

这直接影响 OpenClaw 的回复投递方式：

- 不适合做过于碎片化的流式逐条推送
- 更适合缓冲后合并成较少的文本块再发送
- 需要对超长文本做字节级分片，但也要避免消耗过多消息额度

### 4.3 客户信息接口也有时间窗口

客户基础信息接口并不是随时都能查任意用户。当前文档明确要求：

- 仅支持最近 48 小时内进入过会话或发过消息的客户
- 否则会返回 `invalid_external_userid`

因此，客户画像增强可以做，但应该作为会话中实时补充能力，而不是长期离线全量同步方案。

## 5. 后台前置条件

官方文档和当前整理结果表明，以下后台配置是实际接入的关键前提：

1. 企业已经在微信客服管理后台开通并启用 API
2. 已将某个自建应用配置为“可调用接口的应用”
3. 具体客服账号已授权给该应用
4. 客服账号对应的接待成员处于该应用可见范围内
5. 回调服务同时支持 `HTTP GET` 和 `HTTP POST`
6. 插件侧实现 `access_token` 缓存与失效重取

如果这些条件未满足，代码即使完成，接口也可能无法真正接管消息。

## 6. 与 OpenClaw 架构的匹配关系

从现有的 OpenClaw 架构分析看，微信客服非常适合以“完整渠道插件”的方式接入，而不是修改宿主核心。

推荐的落地方式：

- 插件目录：`extensions/wecom-kf/`
- 渠道 ID：`wecom-kf`
- 配置入口：`channels.wecom-kf`
- 多账号入口：`channels.wecom-kf.accounts.<accountId>`

与 OpenClaw 的主要映射关系如下：

- 微信客服账号：映射为渠道 account
- `external_userid`：映射为 OpenClaw 的 direct peer / target
- `open_kfid`：作为发送侧默认身份或路由维度
- 回调入口：由 `ChannelPlugin.gateway.startAccount` 注册和托管
- OpenClaw 回复：通过 `channel.reply` 和 `channel.routing` 接入统一 Agent 回复链路

换句话说，微信客服在宿主中应表现为一个“新的 DM 渠道”，而不是一个独立业务系统。

## 7. 首版范围为什么应当收敛

从官方文档来看，微信客服还包含客服账号管理、接待人员管理、会话分配、媒体发送、入口生成、第三方代运营等大量扩展能力。

但对 OpenClaw 来说，第一阶段并不需要全做。真正决定“渠道是否可用”的只有消息闭环：

- 能收到用户消息
- 能路由给 Agent
- 能把结果发回去

因此首版应聚焦：

1. 单账号
2. 文本入站
3. 文本出站
4. 欢迎语
5. 基础运行状态

其余能力统一后置，避免在第一版同时解决过多平台侧复杂性。
