# 微信客服插件当前实现状态

## 1. 说明

本文用于总结当前仓库中的 `wecom-kf` 插件已经实现了哪些能力、还缺少哪些能力。

这里描述的是“当前代码实际状态”，不是最初计划中的理想范围。

当前结论基于以下实现：

- `extensions/wecom-kf/`
- `packages/channels/`
- `packages/shared/src/cli/`

## 2. 总体结论

当前 `wecom-kf` 已经完成了首版最小闭环：

1. 外部微信用户通过微信客服发来文本消息
2. OpenClaw 通过回调通知 + `sync_msg` 拉到真实消息
3. 消息进入 Agent 路由与 reply 管线
4. Agent 生成的文本回复再通过微信客服接口回发给用户

也就是说：

- 基础对话能力：已具备
- 文本入站：已具备
- 文本出站：已具备
- 欢迎语：已具备
- 基础状态与探测：已具备

但当前仍然是一个“首版基础渠道”，不是完整的微信客服平台接入。

## 3. 已实现能力

### 3.1 插件与配置接入

已实现：

- 独立插件包：`extensions/wecom-kf/`
- 渠道 ID：`wecom-kf`
- 接入聚合包：`packages/channels`
- CLI 配置向导已支持 `WeCom KF（微信客服）`
- 安装提示已支持 `wecom-kf`
- 配置入口使用 `channels.wecom-kf`
- 配置 schema 已包含：
  - `enabled`
  - `webhookPath`
  - `corpId`
  - `corpSecret`
  - `openKfId`
  - `token`
  - `encodingAESKey`
  - `welcomeText`
  - `dmPolicy`
  - `allowFrom`
  - `defaultAccount`
  - `accounts`

说明：

- 当前首版以单账号为主
- 但配置模型已经预留了 `accounts/defaultAccount`

### 3.2 鉴权与协议层

已实现：

- `corpId + corpSecret -> access_token`
- `access_token` 缓存
- `access_token` 失效后自动重取
- `sync_msg` 封装
- `send_msg` 封装
- `send_msg_on_event` 封装
- 回调 GET 校验
- 回调 POST 验签
- XML / JSON 回调体解密与解析
- 从回调中提取 `token` 与 `open_kfid`

### 3.3 消息拉取与去重

已实现：

- 微信客服回调只做通知，真实消息通过 `sync_msg` 拉取
- `cursor` 持久化
- 短期 `msgid` 去重
- 冷启动“推进游标、不回放历史消息”
- 重启后游标恢复

说明：

- 状态文件当前持久化在本地 `~/.openclaw/wecom-kf/data/state.json`

### 3.4 对话能力

已实现：

- 仅支持 `direct` 会话
- 仅支持客户文本消息入站
- `external_userid` 映射为 OpenClaw 的 DM peer
- 接入 OpenClaw 的 `resolveAgentRoute`
- 接入 `dispatchReplyWithBufferedBlockDispatcher`
- 记录 inbound session 和最近路由
- 支持基础多轮会话延续
- 支持 `dmPolicy`
- 支持 `allowFrom`

当前对话投递策略：

- 不做细粒度流式逐段发送
- 会把 reply 管线里的文本块缓冲后合并
- 对超长文本做 2048 字节分片
- 忽略不可见回复内容

### 3.5 出站发送

已实现：

- 文本回复回发
- Markdown 纯文本降级
- 2048 字节切分
- 按 `open_kfid` 发送
- 支持主动文本发送辅助函数
- 识别目标格式：
  - `user:<external_userid>`
  - `wecom-kf:user:<external_userid>`
  - `user:<external_userid>@<accountId>`

### 3.6 事件能力

已实现：

- `enter_session` 欢迎语
- `msg_send_fail` 事件记录错误摘要

当前只对少量事件做了业务处理，未覆盖微信客服全部事件类型。

### 3.7 运维与可观测性

已实现：

- 基础 onboarding 状态输出
- 基础 probe 能力
- 账号配置状态输出
- webhookPath 输出
- `hasCursor` 状态输出
- 最近入站时间
- 最近出站时间
- 最近同步时间
- 最近欢迎语发送时间
- 最近错误摘要

### 3.8 测试

已实现测试覆盖：

- 回调 GET 校验
- `sync_msg` 分页
- 冷启动游标推进
- 去重
- 文本分片
- 欢迎语事件流程
- CLI 配置向导写入

## 4. 部分实现但还不能算完整能力

### 4.1 多账号

当前状态：部分预留，未作为正式能力完成

已经有的部分：

- 配置 schema 支持 `accounts`
- 存在 `defaultAccount`
- target 格式支持 `@accountId`
- `config.listAccountIds/resolveAccount` 已实现
- onboarding 代码已能写入指定账号配置

仍然缺的部分：

- 没有按“正式多账号场景”做系统联调和验收
- 缺少多账号运维文档
- 缺少多账号专项测试
- 当前实施文档和默认使用路径仍按单账号为主

结论：

- 代码结构已预留多账号
- 但当前不应把它当成稳定完成的多账号能力

### 4.2 对话能力增强

当前状态：只完成了“文本 DM 最小闭环”

已经有的部分：

- 文本入站
- 文本出站
- 会话记录
- Agent 路由

仍然缺的部分：

- 图片消息入站
- 语音消息入站
- 文件消息入站
- 富消息出站
- 卡片类消息
- 更细粒度的事件型交互
- 更完整的客服会话语义支持

## 5. 当前未实现的能力

### 5.2 定时任务与后台作业

## 7. 下一阶段建议优先级

如果继续推进，建议优先级如下：

### P1

1. 多账号正式化
2. 获取客服账号列表
3. 获取客服账号链接
4. 获取客户基础信息
5. 更完整的 probe / status / 错误诊断

### P2

1. 图片 / 文件发送
2. 图片 / 语音 / 文件入站处理
3. 媒体上传与持久化
4. 更丰富的事件处理

### P3

1. 定时任务与主动触达
2. 会话分配策略
3. 接待人员同步
4. 第三方服务商模式

## 8. 一句话结论

当前 `wecom-kf` 已经是一个可用的“微信客服文本 DM 渠道”，但还不是一个完整覆盖微信客服平台能力的插件。

它已经完成了首版最关键的消息闭环，但媒体、定时任务、多账号正式化、客户资料与平台管理能力仍然需要后续实现。
