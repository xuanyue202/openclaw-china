# 微信客服接入 OpenClaw 的开发计划

## 1. 计划目标

本计划用于指导 `wecom-kf` 渠道的第一阶段实现，目标是尽快打通“外部微信用户 -> 微信客服 -> OpenClaw Agent -> 微信客服回复”的最小闭环。

第一阶段完成标准不是覆盖微信客服全部能力，而是让渠道在 OpenClaw 中稳定可用。

## 2. 实施范围

### 2.1 P0 范围

首版必须完成的能力：

1. 新增 `wecom-kf` 渠道插件
2. `channels.wecom-kf` 配置模型
3. `access_token` 获取、缓存、失效重试
4. 回调 GET 验证、POST 验签与解密
5. `sync_msg` 拉取消息
6. `cursor` 持久化与消息去重
7. 文本消息入站
8. 文本消息出站
9. `enter_session` 事件欢迎语
10. 基础状态与日志

### 2.2 P1 范围

第一阶段完成后可以补充：

1. 获取客服账号列表
2. 获取客服账号链接
3. 获取客户基础信息
4. 单图 / 文件类媒体发送
5. 多账号配置
6. 更完整的运行状态与探测

### 2.3 P2 范围

暂时不纳入首版：

1. 客服账号 CRUD
2. 接待人员自动同步
3. 会话分配策略
4. 全量媒体收发
5. 第三方服务商模式
6. 帮助其他企业管理微信客服

## 3. 推荐实现路径

建议采用现有 OpenClaw 架构分析中推荐的“完整插件”方式实现，优先参考 Feishu 风格，而不是复用现有 `wecom` / `wecom-app` 代码结构。

建议目录：

```text
extensions/wecom-kf/
├── index.ts
├── openclaw.plugin.json
├── package.json
└── src/
    ├── channel.ts
    ├── runtime.ts
    ├── types.ts
    ├── config.ts
    ├── crypto.ts
    ├── api.ts
    ├── webhook.ts
    ├── dispatch.ts
    ├── bot.ts
    ├── send.ts
    ├── probe.ts
    └── onboarding.ts
```

## 4. 分阶段任务

### 阶段 1：配置与插件骨架

目标：让 OpenClaw 识别 `wecom-kf` 渠道。

任务：

1. 新建 `extensions/wecom-kf/`
2. 增加 `openclaw.plugin.json`
3. 在 `index.ts` 中注册 `api.registerChannel({ plugin })`
4. 在 `channel.ts` 中定义 `meta`、`capabilities`、`config`、`configSchema`
5. 将统一渠道包 `packages/channels` 纳入 `wecom-kf`

产出：

- OpenClaw 能识别 `wecom-kf`
- `channels.wecom-kf` 配置结构合法

### 阶段 2：鉴权与 API 封装

目标：完成企业微信微信客服 API 的基础访问层。

任务：

1. 实现 `corpId + corpSecret -> access_token`
2. 增加 token 缓存
3. 实现失效后重取逻辑
4. 封装 `sync_msg`
5. 封装 `send_msg`
6. 封装 `send_msg_on_event`
7. 预留媒体上传接口封装

产出：

- 插件具备调用微信客服核心接口的能力

### 阶段 3：回调与消息拉取

目标：完成 webhook 接入和增量消息消费。

任务：

1. 实现 GET 回调校验
2. 实现 POST 验签与解密
3. 从回调正文解析 `token` 与 `open_kfid`
4. 调用 `sync_msg` 拉真实消息
5. 实现 `cursor` 存储
6. 实现重启后的游标恢复
7. 实现消息去重
8. 明确冷启动策略

产出：

- 微信客服回调进入 OpenClaw 后，能够稳定拉到真实消息

### 阶段 4：OpenClaw 入站对接

目标：把微信客服消息接入 OpenClaw 的统一路由与回复系统。

任务：

1. 定义 `external_userid -> direct peer` 映射
2. 构建标准化 inbound context
3. 调用 `resolveAgentRoute`
4. 接入 `dispatchReplyWithBufferedBlockDispatcher`
5. 记录会话状态和最近路由
6. 处理基础 DM policy / allowFrom

产出：

- 外部微信用户消息可以进入 Agent 处理链路

### 阶段 5：出站与欢迎语

目标：将 OpenClaw 的回复回发给微信用户。

任务：

1. 实现文本发送
2. 处理 2048 字节限制
3. 合并回复块，避免过多占用 48 小时消息额度
4. 处理 `enter_session` 事件欢迎语
5. 处理消息发送失败日志

产出：

- 用户能收到来自 OpenClaw 的正常回复
- 新会话在满足条件时可收到欢迎语

### 阶段 6：验证与运维增强

目标：让渠道进入可调试、可维护状态。

任务：

1. 输出账号配置状态
2. 输出 webhook 注册状态
3. 输出最近入站/出站时间
4. 增加基础错误日志
5. 增加配置说明和样例

产出：

- 具备最基本的排障能力

## 5. 首版配置建议

首版建议至少支持以下配置：

```json
{
  "channels": {
    "wecom-kf": {
      "enabled": true,
      "webhookPath": "/wecom-kf",
      "token": "your-token",
      "encodingAESKey": "your-43-char-encoding-aes-key",
      "corpId": "your-corp-id",
      "corpSecret": "your-app-secret",
      "openKfId": "your-open-kfid",
      "welcomeText": "欢迎咨询，我是你的 AI 客服助手。"
    }
  }
}
```

可后续扩展的字段：

- `accounts`
- `defaultAccount`
- `allowFrom`
- `dmPolicy`
- `inboundMedia`
- `apiBaseUrl`

## 6. 风险与注意事项

### 6.1 平台前置条件风险

最大风险不在代码，而在微信客服后台配置：

- 没有配置“可调用接口的应用”
- 客服账号未授权给应用
- 接待成员不在应用可见范围

这些问题会导致接口调用异常或看似成功但消息无法真正接管。

### 6.2 回复策略风险

微信客服有 48 小时和消息条数限制，OpenClaw 的回复必须做聚合，不能直接照搬高频流式输出策略。

建议：

- 首版只发送最终文本
- 不做细粒度流式逐段发送
- 对长文本做受控分片

### 6.3 冷启动风险

`sync_msg` 是游标式增量接口。若游标丢失，可能会重新扫到历史消息。

建议：

- 将 `cursor` 做持久化
- 首次启动采用“推进游标、不回放历史消息”的冷启动策略
- 保留短期 `msgid` 去重

### 6.4 信息补充能力的边界

客户基础信息接口只适用于最近 48 小时有会话行为的客户，且 `unionid` 还有额外绑定约束。

建议：

- 先把它当作会话增强能力
- 不把它作为首版必要依赖

## 7. MVP 验收标准

满足以下条件即可视为首版可用：

1. OpenClaw 能识别并启动 `wecom-kf` 渠道
2. 回调 GET 校验通过
3. 回调 POST 能正确验签、解密和拉消息
4. 客户文本消息能进入对应 Agent 会话
5. Agent 回复能回发给客户
6. `enter_session` 事件可触发欢迎语
7. 渠道运行状态可观察
8. 重启后不会因游标丢失造成大规模历史消息重放

## 8. 建议的开发顺序

推荐按下面顺序推进，避免过早进入复杂能力：

1. 插件骨架与配置
2. `access_token` 与 API 封装
3. 回调验签与解密
4. `sync_msg` 拉取、游标与去重
5. OpenClaw 入站路由
6. 文本回复发送
7. 欢迎语
8. 状态与文档完善

这个顺序的目标是先把消息闭环打通，再逐步补全体验和运维能力。
